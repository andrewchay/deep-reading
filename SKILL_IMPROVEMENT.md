# Deep Reading Skill 优化方案

## 问题分析

### 根本原因

当前Technical Reading Workflow的问题：

```
现行流程:
1. Detect language
2. Identify material type
3. Extract high-level structure
4. Map technical topics
5. Create main report ← 此时链接指向不存在的文件！
6. Create technical modules
7. Establish cross-links
8. Update knowledge base
```

**核心问题**: 第5步创建主报告时，第6步的技术模块还未创建，导致主报告中的WikiLinks指向空文件。

### 具体表现

1. **主报告 → 技术模块**: 链接指向未创建的模块
2. **技术模块 → 其他技术模块**: 如果模块间有引用，先创建的模块无法链接后创建的模块
3. **主报告 → 概念/作者页**: 这些页面可能在知识库更新阶段才创建

---

## 优化方案

### 核心原则

**"先创建被链接的文件，再创建包含链接的文件"**

### 优化后的工作流程

```
阶段1: 分析规划 (Analysis & Planning)
├── 1.1 检测语言和材料类型
├── 1.2 提取内容结构
├── 1.3 确定技术模块清单 (列出所有需要创建的模块)
├── 1.4 确定概念/作者清单
└── 1.5 创建文件夹结构

阶段2: 基础设施 (Infrastructure)
├── 2.1 更新阅读报告索引 (添加占位条目)
└── 2.2 检查并更新MOCs (准备连接点)

阶段3: 技术模块创建 (Technical Modules) ← 先创建叶子节点
├── 3.1 按依赖顺序排序模块 (被依赖的先创建)
├── 3.2 创建所有技术模块 (不包含相互链接)
└── 3.3 在每个模块中添加返回主报告的链接

阶段4: 知识库页面 (Knowledge Base)
├── 4.1 创建/更新作者页面
├── 4.2 创建/更新概念页面
└── 4.3 确保所有概念页面存在

阶段5: 主报告创建 (Main Report) ← 此时所有链接目标已存在
├── 5.1 创建主报告，包含指向已存在模块的链接
├── 5.2 添加延伸阅读链接
└── 5.3 添加指向作者/概念的链接

阶段6: 交叉链接完善 (Cross-Linking)
├── 6.1 在技术模块之间添加相互引用
├── 6.2 更新索引中的条目 (从占位符变为正式链接)
└── 6.3 最终验证所有链接
```

## 具体实施规则

### 规则1: 链接有效性验证

**创建任何文件前，先确认被链接的文件已存在**

```python
# 伪代码示例
def create_link(source_file, target_file, link_text):
    if not file_exists(target_file):
        # 如果目标不存在，先创建目标或移除链接
        if should_create_target(target_file):
            create_placeholder_or_full(target_file)
        else:
            return f"待创建: {link_text}"  # 使用占位符文本
    return f"[[{target_file}|{link_text}]]"
```

### 规则2: 链接层级策略

| 链接方向 | 创建顺序 | 示例 |
|---------|---------|------|
| 主报告 → 技术模块 | 技术模块先 | 主报告中的 `[[技术模块\|详情]]` |
| 技术模块 → 主报告 | 主报告后 | 模块中的 `[[主报告\|返回]]` |
| 模块A → 模块B | 按依赖排序 | 如果A依赖B，先创建B |
| 任何文件 → 概念/作者 | 概念/作者先 | 在阶段4创建 |

## 改进后的技术阅读报告模板

### 主报告模板 (阶段5使用)

```markdown
# {标题} - 技术阅读报告

> **来源**: [[Technical Notes/{标题}/{模块A}\|{模块A}]], [[Technical Notes/{标题}/{模块B}\|{模块B}]]

... (其他内容)

## 四、技术细节模块索引

- 📁 [[Technical Notes/{标题}/{模块A}\|{模块A}]]
- 📁 [[Technical Notes/{标题}/{模块B}\|{模块B}]]
```

### 技术模块模板 (阶段3使用)

```markdown
# {标题} - {模块名}

> **来源**: [[{标题} - 技术阅读报告\|主报告]]  
> **相关概念**: [[概念A]], [[概念B]] (阶段4后填充)

## 内容...

---

## 与其他模块的联系

- **前置**: [[模块X]] (如果有)
- **后续**: 待补充
```

## 实施检查清单

### 阶段1: 分析规划
- [ ] 列出所有需要创建的技术模块
- [ ] 列出所有涉及的概念
- [ ] 列出所有涉及的作者
- [ ] 确定模块依赖关系
- [ ] 创建文件夹结构

### 阶段2: 基础设施
- [ ] 在阅读报告索引中添加占位条目

### 阶段3: 技术模块
- [ ] 按依赖顺序创建所有模块
- [ ] 每个模块只包含返回主报告的链接
- [ ] 不包含指向其他模块的链接（留待阶段6）

### 阶段4: 知识库
- [ ] 创建/更新所有作者页面
- [ ] 创建/更新所有概念页面

### 阶段5: 主报告
- [ ] 验证所有链接目标已存在
- [ ] 创建主报告，包含所有链接

### 阶段6: 交叉链接
- [ ] 在技术模块之间添加相互引用
- [ ] 更新索引中的占位符为正式链接
- [ ] 最终验证：搜索所有 `[[...]]` 确保目标存在

---

## 预期效果

1. **零空链接**: 所有WikiLinks都指向已存在的文件
2. **更好的导航**: 用户可以顺畅地在报告、模块、概念之间跳转
3. **知识图谱完整性**: 知识库的连接更加完整和有用
4. **减少维护成本**: 不需要后续手动修复断链

---

## 应用到现有报告

对于已经生成的报告，建议：
1. 批量检查所有技术报告中的链接
2. 对空链接进行回填或标记
3. 建立链接检查脚本/工具
